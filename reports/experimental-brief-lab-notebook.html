Release 1 · Experimental Brief

Can PageIndex make a personal website corpus more legible to an AI system?

A bounded, reproducible experiment using a curated Markdown representation of a Jekyll-based website. It separates two frequently conflated AI tasks: building a map of knowledge and reasoning over that map.

Primary corpusOne normalized Markdown “site book”
Index conditionsDeterministic, cloud, local
Retrieval conditionsOpenAI, Claude, Ollama
Question set14 questions across 5 types (frozen)

Purpose and research framing

Central research question: How do source structure, index-generation model, and retrieval model affect what an AI system can accurately recover, connect, and explain about a curated body of professional work?

Personal value

Discover where the public site communicates a coherent professional identity, where it contains contradictions, and where important ideas are present but hard to find.

Technical value

Separate preprocessing, indexing, retrieval, and generation effects while testing hierarchical, reasoning-based RAG across providers.

Publication value

Produce narrow, evidence-based claims about knowledge architecture rather than a generic “new RAG tool worked” demonstration.

Claims this experiment may support
  • PageIndex was more effective for a particular class of cross-section question.
  • A deterministic hierarchy matched or outperformed model-generated summaries on a structured source.
  • A capable model navigated a strong index even when it did not build that index.
  • A local model was viable for one PageIndex role but not the other.
  • Knowledge representation mattered as much as model choice.
Claims this experiment does not justify by itself
  • “PageIndex is better than RAG.”
  • “Local models are as good as cloud models.”
  • “This method scales to an enterprise corpus.”
  • “One strong answer proves reliable retrieval.”

First recommended release

Intentionally narrow: establish the harness and test the most informative variables before adding PDF rendering, vector baselines, or multi-document routing.

Scope guardrail: Do not begin with a full three-by-three factorial. First hold the index constant to compare retrievers; then hold the retriever constant to compare index builders.

Corpus design

The corpus should behave like a carefully edited “site book,” not a raw dump of Jekyll files.

Evaluation rubric

Score each layer independently. A poor answer does not establish whether preprocessing, indexing, retrieval, or synthesis failed.

A. Preprocessing

  • Content completeness
  • Boilerplate removal
  • Heading fidelity
  • Metadata fidelity
  • Provenance

B. Index quality

  • Structural coverage
  • Correct hierarchy
  • Boundary accuracy
  • Summary faithfulness
  • Duplicates or hallucinations

C. Retrieval quality

  • Correct branch
  • Evidence completeness
  • Unnecessary retrieval
  • Recovery from a miss
  • Tool-call efficiency

D. Answer quality

  • Correctness
  • Completeness
  • Grounding
  • Uncertainty
  • Usefulness

E. Operational behavior

  • Index duration
  • Query latency
  • Tool calls / query (navigation effort)
  • Structure tokens / query (index navigation cost — the IDX-D vs IDX-C headline)
  • Content tokens / query (retrieval tightness)
  • Total tokens & estimated cost / query
  • Errors and retries
  • Run stability

Suggested scale

0 — failed / absent1 — materially poor2 — mixed / partial3 — strong4 — excellent

Run protocol

Recommended sequence

  1. Harness validation: one index, one retriever, three questions.
  2. Retriever comparison: hold one index fixed.
  3. Index comparison: hold one retriever fixed.
  4. Selected cross-pairs: test only combinations that answer a substantive question.
Stop condition: If the harness cannot preserve raw outputs, traces, tree versions, and configuration metadata, pause before producing comparisons.

Next Steps and Why

What follows Release 1

1. Render selected HTML pages as PDFs

Why: Compare an author-explicit Markdown hierarchy with a model-inferred hierarchy from a visual document representation.

2. Add a conventional retrieval baseline

Why: Determine whether PageIndex helps on particular query classes rather than evaluating it in isolation.

3. Test multiple documents and routing

Why: Separate within-document navigation from the harder problem of choosing which document to inspect.

4. Automate preprocessing

Why: Turn a one-off export into a reproducible content-to-corpus pipeline with provenance and validation.

5. Evaluate repeated runs

Why: Distinguish robust behavior from one persuasive but lucky trace.

6. Shape the publication narrative

Why: Let evidence determine whether the piece is about PageIndex, local models, information architecture, or knowledge legibility.

Likely thesis: AI-readiness is not only a model-selection problem. It is also a representation problem: hierarchy, provenance, and boundaries influence what an AI system can reliably discover.

Open questions before implementation

  1. Which Ollama model should be nominated for index summaries, and which local model should be tested for tool-based retrieval?
  2. Which capable cloud model should serve as the reference index builder?
  3. Should the first corpus emphasize professional positioning, technical projects, or a balanced representation?
