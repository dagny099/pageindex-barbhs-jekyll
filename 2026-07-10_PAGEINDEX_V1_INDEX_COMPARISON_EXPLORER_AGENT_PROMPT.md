# Build a PageIndex V1 Index Comparison Explorer

**Date:** 2026-07-10  
**Purpose:** Repo-local coding-agent prompt  
**Primary output:** `reports/V1_INDEX_COMPARISON.html`

You are working inside the local repository for the PageIndex website experiment.

Your task is to build a **self-contained, static Index Comparison Explorer** that makes the three V1 JSON indexes visually inspectable and genuinely useful for human review.

The raw JSON files are technically complete but difficult to understand. The explorer should present them as synchronized document maps rather than as JSON blobs.

This is an inspection and comparison tool, not a results dashboard and not a new retrieval system.

---

# Experiment context

The experiment uses one frozen Markdown corpus and three index conditions:

- **IDX-D — Deterministic Markdown Index**  
  The hierarchy comes from the authored Markdown headings. It has no model-generated node summaries.

- **IDX-C — Cloud-Assisted Index**  
  The same Markdown hierarchy is enriched with summaries generated by a cloud model.

- **IDX-O — Ollama-Assisted Index**  
  The same Markdown hierarchy is enriched with summaries generated by a local Ollama model.

The main V1 question is:

> Do generated summaries make the document map meaningfully easier to navigate than the authored Markdown hierarchy alone?

The explorer should help inspect:

- structural fidelity;
- missing or duplicated nodes;
- hierarchy differences;
- node boundaries;
- summary faithfulness;
- summary specificity;
- repeated or generic summaries;
- contradictions or distinctions erased by summaries;
- differences between IDX-C and IDX-O;
- whether raw node text corresponds to the expected source section.

# Alignment strategy

The three indexes are expected to represent substantially the same hierarchy, but node IDs may not be stable.

Implement alignment conservatively.

Prefer, in order:

1. Exact source path plus normalized heading path.
2. Exact normalized heading path plus matching line range.
3. Source path plus normalized title plus parent path.
4. Normalized title plus strongly overlapping boundaries.
5. Explicitly unmatched.

Do not align nodes merely because their titles match.

Titles such as:

```text
How It Works
Summary
Why This Matters
Architecture
```

occur repeatedly and must be disambiguated by ancestry and source context.

Normalize for comparison carefully:

- Unicode normalization;
- trim whitespace;
- collapse repeated whitespace;
- ignore cosmetic Markdown link syntax where appropriate;
- preserve meaningful punctuation and numbers;
- do not conflate different headings aggressively.

Assign an alignment status:

- exact;
- high confidence;
- probable;
- unmatched;
- ambiguous.

Record every non-exact match in the alignment report.

Never silently drop unmatched nodes.

# Difference detection

The “Show differences only” mode should surface nodes where any of the following apply:

- node missing from one condition;
- title differs;
- parent path differs;
- child count differs;
- line or page boundaries differ;
- summary missing unexpectedly;
- IDX-C and IDX-O summaries differ substantially;
- summary is unusually short;
- summary is unusually long;
- summary is nearly identical to many other summaries;
- summary appears generic;
- summary contains numbers absent from the source section;
- summary mentions a project, status, technology, or claim absent from the source;
- alignment is ambiguous.

For automatic semantic-faithfulness warnings, use conservative deterministic heuristics only.

Do not call an LLM during report generation.

Safe heuristics include:

- named numbers in summary not found in source;
- repeated summary text across many nodes;
- identical summary across unrelated nodes;
- summary-to-source token overlap extremely low;
- summary contains status terms such as “production,” “deployed,” or “complete” absent from source;
- summary references an entity not present in the source section.

Label these as:

```text
Review signal
```

not as confirmed errors.
